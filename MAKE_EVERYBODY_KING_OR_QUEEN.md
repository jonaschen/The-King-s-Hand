## 先讀這段 — 這份計畫為何存在

The King's Hand 已經證明了一件事：當 AI 停止當一個禮貌的摘要機器，轉而扮演一個冷酷的敵對性分析師時，它能夠從一堆「一切正常」的專案報告裡，刺穿西瓜效應，挖出被掩埋的真相。

但 Henry 不是唯一需要這種能力的人。

一位醫院營運長需要知道，科室主任口中的「人力調配已優化」是否意味著護病比已經超標三個月。一位新創公司 CTO 需要知道，工程週報裡的「技術評估進行中」是否意味著核心架構已經走進死胡同但沒人敢說。一位供應鏈副總需要知道，供應商回覆的「交期略有調整」是否意味著產線即將斷料。

**西瓜效應不是半導體業的專利，它是所有矩陣組織、所有層級彙報結構的宿命。**

但光有分析能力還不夠。

一位真正有用的幕僚，不是你每週丟一疊文件給他、他吐一份報告回來的分析機器。他是一個**持續在你身邊的工作夥伴**。你走進辦公室跟他說「老闆剛給了我一個新專案」，他記住了。你下午跟他說「Android build study 卡住了，找不到 root cause」，他更新了狀態。週五到了，你不需要花兩小時整理週報——他已經從這一週所有的對話中幫你組織好了。三週後他主動提醒你：「那個 build study 三週沒有進展了，要不要在週會上提出來？」

> **現有系統的定位：** 「給我一份文件，我告訴你裡面藏了什麼。」
>
> **本計畫的願景：** 「成為你持續在場的工作夥伴。了解你的世界。追蹤你的現實。在你需要穿透迷霧的時候，拿出敵對性分析這把利刃。」

The King's Hand 的敵對性分析能力不會消失——它會成為這位幕僚最銳利的一項技能。但幕僚的價值不只在於那把劍。他的價值在於每天都在、持續理解你的處境、在你需要之前就準備好了。

本計畫的目標，是讓每一個人都擁有這樣一位幕僚——先認識你，持續陪伴你，在關鍵時刻為你刺穿假象。

## 從 The King's Hand 到通用系統：已驗證的資產盤點

在開始設計新架構之前，必須清楚盤點 The King's Hand 中哪些是可直接複用的核心資產，哪些是需要被抽象化的領域綁定元件。這決定了重構的邊界。

### 可直接複用的核心架構（Domain-Agnostic Core）

| 資產 | 通用價值 | 複用方式 |
|:---|:---|:---|
| **五步推理鏈（5-Step Chain of Thought）** | 解構→定位→矛盾偵測→影響外推→提問合成——這是一套完整的敵對性分析方法論，不依賴任何特定領域 | 保留為推理引擎骨幹，將每一步的具體指令模板化 |
| **五模組報告結構** | 定位→紅綠燈→摩擦分析→市場連動→靈魂三問——這是一個從「這是什麼」到「所以你該問什麼」的完整敘事弧線 | 保留結構，但模組標題與內容指引根據領域動態生成 |
| **紅綠燈覆寫機制** | 「技術證據永遠凌駕官方報告狀態」——這是系統的靈魂，適用於任何存在資訊過濾的組織 | 直接複用，覆寫觸發條件由領域框架定義 |
| **敘事體寫作規則** | 禁止分析模組使用項目符號；強制敘事體散文——這迫使系統進行深度綜合而非條列式偷懶 | 直接複用 |
| **靈魂三問類型學** | 防禦/問責、規模外推、生態護城河——三個角度涵蓋了「眼前危機」「未來傳染」「戰略位置」 | 保留三類型框架，具體問題模板由領域知識填充 |
| **Step 0 自適應推理（ROADMAP v1.6 已實作）** | 領域推斷 → 框架建構 → 異常信號擴展——已經證明了零配置啟動的可行性 | 作為通用化的起點，大幅擴展 |
| **CALIBRATE.md 校準元技能** | 從對話記錄中萃取管理者輪廓——已經證明了漸進式學習的可行性 | 擴展為完整的偏好學習系統 |

### 需要抽象化的領域綁定元件（Domain-Specific → Domain-Parameterized）

| 原始元件 | IC 設計中的具體實現 | 抽象化後的通用框架 |
|:---|:---|:---|
| `software_stack_map.md`（5 層軟體堆疊） | Bootloader → BSP → OS → Middleware → SDK | **價值鏈地圖（Value Chain Map）**：任何領域都有從底層基礎到上層價值的分層結構 |
| `workaround_taxonomy.md`（8 種 Workaround） | Polling spinlock, NOP insertion, data throttling... | **補丁模式庫（Patch Pattern Library）**：任何領域都有「用短期修補掩蓋結構性問題」的模式 |
| `corporate_code_words.md`（25 種企業話術） | "Ongoing integration with minor delays" → HW/SW 介面未凍結 | **術語煙幕辭典（Euphemism Dictionary）**：任何組織都有將壞消息包裝成中性語言的慣用語 |
| `rag_status_criteria.md`（紅綠燈規則） | Tape-out delay → 🔴, Customer escalation → 🔴 | **覆寫觸發規則（Override Trigger Rules）**：由「硬性約束」和「不可逆里程碑」定義 |
| PPA 量化框架 | Power / Performance / Area | **取捨軸量化框架（Trade-off Axes）**：任何領域都有其核心取捨維度 |

### Step 0 已建立的六元素領域框架

ROADMAP v1.6 的 Self-Adjustment Addendum 中，Step 0 定義了以下六個領域無關的分析元素，這是通用化的基石：

1. **Hard Constraint（硬性約束）**：該領域中不可違反的絕對限制（IC 設計中是物理定律與製程限制；醫療中是法規與患者安全；金融中是合規要求）
2. **Patch Pattern（補丁模式）**：該領域中「短期修補代替結構性修復」的典型手法
3. **Irreversible Milestone（不可逆里程碑）**：一旦錯過就無法挽回的關鍵節點（tape-out、產品上市、法規申報截止）
4. **Trade-off Axes（取捨軸）**：該領域的核心資源取捨維度（PPA、成本-品質-速度、風險-報酬）
5. **Tier-1 Relationship（一級關係）**：最重要的外部利害關係人（客戶、投資人、監管機構）
6. **Blame Transfer Seam（責任轉嫁接縫）**：組織中最容易發生責任推諉的分工邊界

**這六個元素構成了通用領域框架的完整參數空間。任何領域的分析，都可以被映射到這六個維度上。**

## 系統演進願景：三層自適應架構

基於 The King's Hand 的已驗證資產，以及「AI 助手進化開發計畫」中提出的進階架構研究，本計畫定義三層演進架構。每一層獨立可用，但層層疊加後產生倍增效果。

```
┌─────────────────────────────────────────────────────────────┐
│  第三層：主動式智慧（Proactive Intelligence）                │
│  ─ 趨勢偵測、期限預警、週報自動生成                          │
│  ─ 不等你開口，在問題浮現前主動介入                           │
├─────────────────────────────────────────────────────────────┤
│  第二層：分析兵器庫（Analytical Arsenal）                     │
│  ─ 敵對性分析、矛盾偵測、靈魂三問                            │
│  ─ The King's Hand 的核心能力，作為幕僚的專精技能              │
├─────────────────────────────────────────────────────────────┤
│  第一層：持續在場的夥伴（Persistent Partner Foundation）      │
│  ─ 初識對話、活的工作狀態、日常更新吸收                       │
│  ─ 每天都在你身邊，理解你的世界，追蹤你的現實                  │
└─────────────────────────────────────────────────────────────┘
```

**與前一版架構的根本差異：**

前一版的第一層是「通用敵對性分析核心」——系統的基礎身份是一個分析師，你丟文件給他，他回報告。持續進化和主動智慧是疊加在分析之上的增強。

新架構反轉了這個關係。**第一層是「持續在場」本身**——幕僚的基礎身份是你的工作夥伴，他每天都在、持續追蹤你的狀態、吸收你的日常更新。敵對性分析是他在第二層被裝備的一把武器，在需要的時候抽出來用。

這個反轉不是語意上的重新包裝。它改變了系統的核心資料流：

| 維度 | 前一版（分析師模型） | 新架構（夥伴模型） |
|:---|:---|:---|
| **主要輸入** | 文件包（週報、Jira、會議記錄） | 日常對話流（「老闆給了新任務」「build study 卡住了」「客戶回信了」） |
| **主要輸出** | 五模組分析報告 | 持續更新的工作狀態 + 按需觸發的分析報告 + 自動生成的週報 |
| **互動頻率** | 每週一次（會前分析） | 每天多次（隨時更新） |
| **核心狀態** | 無狀態（每次分析獨立） | 有狀態（維護活的 TODO、專案狀態、待辦追蹤） |
| **分析觸發** | 使用者主動提交文件 | 幕僚從日常對話中偵測到需要深度分析的信號，或使用者主動要求 |

## 第一層：持續在場的夥伴（Persistent Partner Foundation）

### 設計目標

使用者不是每週來找幕僚一次。他們每天都跟幕僚說話——有時候是交代一件新任務，有時候是回報一個進展，有時候只是抱怨一下今天的會議很浪費時間。幕僚聽著、記住、追蹤，在需要的時候提供判斷。

**最低可行體驗：** 一個工程主管在週一告訴系統「老闆指派了一個新的 Android build study」，週二說「找到一個可能的 root cause 但還需要驗證」，週四說「驗證失敗，回到原點」。到了週五，系統能自動生成一段週報摘要，準確反映這個任務的狀態——而不需要主管從零開始寫。

### 核心元件設計

#### 1. 初識對話（First-Day Dialogue）

（此元件的設計已在 Step 0 雙模態設計中詳細定義。）

一位幕僚到任第一天做的事：坐下來聊，了解主人的角色、團隊、工具、焦慮、判斷標準。這場對話不只建構六元素框架——它建立的是幕僚對主人整個工作世界的基礎理解。

初識對話完成後，系統擁有：
- **Manager Profile v1**：角色、領域、團隊結構、關注重點、溝通風格
- **六元素框架 v1**：Hard Constraint、Patch Pattern、Irreversible Milestone、Trade-off Axes、Tier-1 Relationship、Blame Transfer Seam
- **初始工作狀態**：主人目前手上有哪些事（從對話中擷取）

#### 2. 活的工作狀態（Living Work State）

這是整個系統最核心的新概念。The King's Hand 是無狀態的——每次分析獨立，不記得上次分析了什麼。持續在場的夥伴必須是有狀態的——他隨時知道你手上有哪些事、每件事目前的進展、哪些在動、哪些卡住了。

**Living Work State 的資料結構：**

```markdown
## 工作狀態：{使用者名稱}
最後更新：2026-03-22 14:30

### 進行中的專案與任務

| ID | 項目 | 狀態 | 最後更新 | 靜止天數 | 備註 |
|:---|:---|:---|:---|:---|:---|
| T-001 | Android build study | 🔴 卡住 | 03-20 | 2 | root cause 驗證失敗，回到原點 |
| T-002 | Q2 HC planning (25%) | 🟡 進行中 | 03-22 | 0 | 等待 HR 回覆 headcount ceiling |
| T-003 | NPU compiler benchmark | 🟢 正常 | 03-21 | 1 | 跑分結果符合預期 |
| T-004 | 跨部門 API 標準化提案 | ⚪ 等待中 | 03-15 | 7 | 等待架構委員會排審 |

### 近期關鍵事件
- 2026-03-22：老闆在週會上追問 build study 進度，承諾下週給答覆
- 2026-03-20：build study root cause 驗證失敗
- 2026-03-18：收到 Q2 HC planning 任務，deadline 04-05

### 待追蹤的承諾
- 下週一前給老闆 build study 更新 → 3 天後
- HR headcount ceiling 回覆 → 等待中
```

**Living Work State 的維護機制：**

| 觸發 | 系統行為 |
|:---|:---|
| 使用者說「老闆給了一個新任務」 | 建立新的任務條目，推斷優先級和 deadline |
| 使用者說「build study 有進展了」 | 更新對應任務的狀態和最後更新時間，重置靜止天數 |
| 使用者說「那個 API 提案可以先放著」 | 將任務狀態改為 ⚪ 等待中 |
| 使用者說「benchmark 跑完了，結果 OK」 | 將任務標記為 ✅ 完成，移入歷史 |
| 使用者丟進一份文件要求分析 | 觸發第二層敵對性分析，分析結果關聯到對應的任務 |
| 幕僚偵測到某任務靜止天數 > 閾值 | 在下次對話中自然提及（不是推播通知，是對話中的提醒） |

**關鍵設計原則：自然語言輸入，結構化輸出。** 使用者不需要用任何特定格式跟幕僚說話。「早上開會老闆又追問 build study 了」「HR 還沒回」「benchmark OK 了」——這些都是自然對話。系統在 `<thinking>` 中將這些對話解析為狀態更新，維護結構化的 Living Work State。使用者看到的永遠是一位在聽你說話的幕僚，不是一個要求你填表格的系統。

#### 3. 日常更新吸收（Daily Update Ingestion）

使用者和幕僚的日常對話，本質上是一系列狀態更新。系統需要能夠從自然對話中準確擷取以下資訊：

| 資訊類型 | 範例對話 | 系統擷取 |
|:---|:---|:---|
| **新任務** | 「老闆剛分配給我一個新的客戶拜訪準備」 | 建立 T-005，狀態 🟢，推斷 deadline |
| **進展更新** | 「API 標準化提案過了架構委員會初審」 | T-004 狀態從 ⚪ 改為 🟡 |
| **阻塞回報** | 「build study 的測試環境掛了，要等 IT 修」 | T-001 備註更新，靜止原因記錄 |
| **完成確認** | 「NPU benchmark 報告交出去了」 | T-003 標記 ✅ |
| **承諾記錄** | 「我跟客戶說下週三前給規格書」 | 新增到「待追蹤的承諾」 |
| **情緒/脈絡** | 「今天那個會真的很浪費時間」 | 不更新任何狀態，但記錄為脈絡——影響幕僚理解主人目前的壓力程度 |

#### 4. 週報與狀態報告生成（Report Generation）

幕僚累積了一週的日常對話後，具備了自動生成週報的能力。這不是從零撰寫——這是從已經持續追蹤的 Living Work State 中，組織出一份結構化的報告。

**報告生成的觸發方式：**
- 使用者主動要求：「幫我整理一下這週的週報」
- 排程提醒：「今天是週五，需要我幫你整理週報嗎？」
- 依使用者偏好的格式和語言輸出

**報告與 The King's Hand 分析的關係：** 週報生成是第一層的基礎能力——它只是組織事實。如果使用者需要對某個專案進行深度的敵對性分析（例如「幫我看看這份 PM 報告有沒有問題」），這會觸發第二層的分析兵器庫。兩者可以結合：週報中包含事實整理，附帶對特定文件的深度分析。

### 第一層驗收標準

| 檢核項目 | 目標 | 驗證方式 |
|:---|:---|:---|
| 初識對話有效性 | 3-5 輪對話後，系統建立的 Manager Profile 準確反映使用者的工作脈絡 | 人工審查 |
| 日常更新吸收準確率 | 從 10 段自然對話中，正確擷取任務狀態更新 ≥ 90% | 比較系統擷取結果與預期結果 |
| Living Work State 一致性 | 經過 5 天模擬互動後，工作狀態表準確反映所有任務的最新狀態 | 人工審查 |
| 週報生成品質 | 從一週的模擬對話中自動生成的週報，涵蓋所有關鍵進展和阻塞 | 人工評估完整性和準確性 |
| 知識厚度自評準確性 | IC 設計文件自評為 L3；陌生領域首次接觸自評為 L1 | 三個厚度等級各至少一個測試案例 |
| IC 設計零退化 | 原有 16 個測試案例的通過率維持 100%（透過第二層觸發） | 回歸測試 |

## 第二層：分析兵器庫與持續進化（Analytical Arsenal & Continuous Evolution）

### 設計目標

第一層讓幕僚持續在場、追蹤現實。第二層給幕僚兩樣東西：**一把利刃**（The King's Hand 的敵對性分析能力）和**越用越聰明的大腦**（從互動中學習的能力）。

利刃不是每天都用的。大多數時候，主人只是在跟幕僚說話、更新狀態。但當主人說「幫我看看這份報告有沒有問題」或者幕僚自己從日常對話中嗅到了危機的前兆時，利刃就要出鞘。

### 分析兵器庫（The King's Hand as a Skill）

The King's Hand 的完整分析能力——五步推理鏈、五模組報告、六元素框架、紅綠燈覆寫、靈魂三問——在新架構中成為幕僚的一項專精技能，而非他的全部身份。

**觸發分析的場景：**

| 觸發方式 | 範例 | 執行內容 |
|:---|:---|:---|
| **使用者主動要求** | 「幫我分析這份週報」「這份 Jira 匯出有沒有問題」 | 完整五模組分析 |
| **使用者帶著具體問題** | 「這封信是不是在掩飾什麼」 | 快速掃描（Step 1 + Step 3 子集） |
| **幕僚從日常對話中偵測信號** | Living Work State 中某個任務持續停滯，且主人的描述語氣從具體轉為模糊 | 幕僚主動建議：「你最近提到 build study 的方式讓我覺得可能有被掩蓋的問題。要不要把相關文件給我看看？」 |
| **週報生成中的附帶分析** | 週報整理過程中發現某個專案的進展描述與 Living Work State 中的實際狀態矛盾 | 在週報中標注矛盾，建議進一步分析 |

**分析結果與 Living Work State 的整合：** 分析不再是孤立的報告。每次分析的關鍵發現會被摘要後寫回 Living Work State，成為該任務/專案的歷史脈絡。下次分析同一個專案時，幕僚能看到上一次的發現，進行趨勢比較。

### 持續進化引擎（Continuous Learning）

幕僚越用越懂你。這不是靠魔法——是靠系統性地從每次互動中學習。

#### 1. 分層記憶架構（Hierarchical Memory Architecture）

參考「AI 助手進化開發計畫」中的 HMLR 架構與事件演進圖概念，但務實地設計為 Claude Agent Skill 生態系統中可實現的版本：

**四層記憶模型：**

| 記憶層 | 存儲內容 | 持久性 | 實作機制 |
|:---|:---|:---|:---|
| **工作記憶（Working Memory）** | 當前對話的上下文 | 會話內 | LLM 上下文視窗（已有） |
| **活的工作狀態（Living Work State）** | 任務清單、專案狀態、待追蹤承諾、近期事件 | 跨會話持久，高頻更新 | `work_state/{user}.md`，每次對話都可能更新 |
| **使用者輪廓（User Profile）** | 領域、職位、關注重點、溝通風格偏好、術語表 | 跨會話持久，低頻更新 | `manager_profiles/{user}.md`，由 CALIBRATE.md 維護 |
| **領域知識庫（Domain Knowledge Base）** | 領域框架、補丁模式庫、術語辭典 | 跨使用者持久 | `domain_packs/{domain}/` 目錄下的結構化文件 |

**Living Work State 與 User Profile 的分離：** 這兩者的更新頻率完全不同。Living Work State 每天都在變——新任務進來、舊任務完成、狀態更新。User Profile 是慢變量——領域不會天天換，溝通偏好不會每次對話都變。分離它們是為了讓高頻更新不干擾低頻知識的穩定性。

**關鍵設計決策：為什麼不直接上圖資料庫？**

「AI 助手進化開發計畫」提出了圖結構記憶與 HMLR 等先進架構。這些在技術上是正確的方向，但在當前階段存在實際約束：Claude Agent Skill 的執行環境以文件系統為主要持久化機制，不具備常駐的圖資料庫服務。因此，本計畫採取「結構化 Markdown 文件 + 語意索引」的務實路線，在概念上模擬分層圖譜的效果：

- **實體追蹤**：在 User Profile 中以結構化表格維護關鍵實體及其狀態變化歷史（模擬節點與邊）
- **時序記憶**：以帶時間戳的事件日誌追蹤分析歷史（模擬時序圖）
- **語意去重**：CALIBRATE.md 在更新 Profile 時，負責合併重複資訊、解決矛盾（模擬園丁機制）

**進階路線圖**：當系統遷移到具備持久化服務的環境（如 MCP Server 或自託管服務）時，可無縫升級為真正的圖結構記憶。結構化 Markdown 是圖資料庫的序列化形式，遷移成本極低。

#### 2. 隱性偏好學習系統（Implicit Preference Learning）

參考 CIPHER 演算法的核心概念，設計為可在 Skill 生態系統中落地的版本：

**偏好萃取觸發時機：**

| 觸發事件 | 萃取內容 | 儲存位置 |
|:---|:---|:---|
| 使用者修改了報告的措辭 | 原始輸出 vs. 修改後結果 → 推斷表達風格偏好 | User Profile → `style_preferences` 區塊 |
| 使用者追問了某個模組的某個方向 | 追問方向 → 推斷關注重點優先序 | User Profile → `priority_order` 區塊 |
| 使用者否定了某個分析結論 | 否定原因 → 推斷領域知識修正 | Domain Knowledge Pack → 對應文件 |
| 使用者提供了系統未見過的文件類型 | 文件結構 → 推斷新的輸入格式處理規則 | Domain Knowledge Pack → `input_formats` |
| 使用者對靈魂三問表示某一類型特別有用 | 問題類型偏好 → 調整三問的配比策略 | User Profile → `question_preferences` 區塊 |

**偏好的透明性原則（White-Box Preferences）：**

所有被推斷的偏好都以自然語言儲存在 User Profile 中，使用者可以隨時檢視、修改或刪除。系統永遠不會建立使用者無法理解或無法控制的隱性模型。這不僅是技術選擇，更是信任的基石。

**CALIBRATE.md 的擴展：**

現有的 CALIBRATE.md 僅從單次對話記錄中萃取 Manager Profile。擴展後的版本將支援：

- **增量更新模式（Incremental Update）**：不再每次重建 Profile，而是比對新舊版本，僅合併新資訊
- **衝突解決邏輯**：當新的互動證據與既有 Profile 矛盾時，依據「近期行為 > 早期行為」「明確聲明 > 推斷偏好」的優先規則解決
- **信心衰減機制**：超過 N 次互動未被再次驗證的偏好，其信心等級逐步降低
- **跨領域遷移**：當偵測到使用者切換了工作領域，將通用偏好（如溝通風格、報告詳細度）遷移到新領域的 Profile 中，但不遷移領域特定知識

#### 3. 技能動態擴充機制（Dynamic Skill Expansion）

系統應能根據互動中遇到的新需求，自動擴充自己的分析能力：

**擴充觸發場景：**

| 場景 | 擴充動作 |
|:---|:---|
| 使用者反覆提交某種系統不擅長處理的文件格式 | 建立新的輸入解析規則，加入 Domain Pack |
| 使用者指出了系統遺漏的某類異常信號 | 將新信號加入領域框架的異常關鍵詞庫 |
| 使用者提供了系統不認識的術語翻譯 | 更新 Euphemism Dictionary |
| 某個領域的分析被反覆使用且品質穩定 | 將動態建構的框架固化為正式的 Domain Knowledge Pack |

**品質閘門（Quality Gate）：**

動態擴充必須通過品質閘門才能被持久化：

1. **一致性檢查**：新知識不得與既有知識庫矛盾（除非使用者明確覆寫）
2. **最小使用次數**：某個推斷偏好或新知識點必須在 ≥ 3 次互動中被使用且未被否定，才會從「候選」升級為「確認」
3. **使用者確認**：重大的知識庫變更（如新增完整的 Domain Pack）會在下次互動時簡要通知使用者並徵求確認

#### 第一層與第二層的結構性關係：知識沉澱迴圈

第二層的所有機制——偏好學習、記憶維護、技能擴充——在概念上做的是同一件事：**把 Bottom-Up 浮現的知識，沉澱為 Top-Down 可用的框架，推動知識厚度從 L1 向 L3 爬升。**

```
互動 1-3（L1 淺薄）
  ├─ Bottom-Up 主導：資料浮現異常模式
  ├─ 系統產出探索性報告
  └─ CALIBRATE.md 從互動中萃取：領域術語、使用者關注點、異常信號

互動 4-8（L2 中等）
  ├─ 候選知識通過品質閘門，升級為確認知識
  ├─ 補丁模式庫開始有具體實例（而非僅抽象描述）
  ├─ 術語辭典覆蓋度上升
  └─ Top-Down 與 Bottom-Up 比例趨近 50:50

互動 9+（邁向 L3 深厚）
  ├─ Domain Pack 累積足夠的已驗證知識
  ├─ 覆寫觸發規則有了量化閾值
  ├─ Top-Down 比例上升，分析精度與效率同步提升
  └─ 系統行為逐漸趨近 The King's Hand 在 IC 設計中的表現
```

**這意味著：通用化的系統不是 The King's Hand 的弱化版——它是 The King's Hand 的幼年形態。** 給它足夠的互動與校準，它在任何領域都能長成和 IC 設計版本同樣鋒利的武器。知識厚度自評（Step 0.5）確保它在成長過程中始終誠實，不會在尚未成熟時假裝自己已經成熟。

### 第二層驗收標準

| 檢核項目 | 目標 | 驗證方式 |
|:---|:---|:---|
| 分析技能觸發準確性 | 使用者要求分析時正確觸發完整五模組報告；帶具體問題時觸發快速掃描；日常對話時不觸發分析 | 模擬不同類型的互動 |
| 分析結果回寫 | 分析的關鍵發現被正確摘要到 Living Work State 的對應任務 | 人工審查 |
| 偏好學習有效性 | 經過 5 次互動後，報告風格與使用者偏好的匹配度顯著提升 | 比較第 1 次與第 5 次輸出 |
| **知識厚度爬升** | 一個 L1 領域在 ≥ 8 次互動後升級為 L2 | 模擬多輪互動的縱向測試 |
| 偏好透明性 | 使用者能在 Profile 中找到所有被推斷的偏好，且描述可理解 | 人工審查 Profile |
| IC 設計分析零退化 | 16 案例回歸通過 | 回歸測試 |

## 第三層：主動式智慧（Proactive Intelligence）

### 設計目標

幕僚不只是被動地等主人來說話。他會在適當的時機主動開口：「那個 build study 已經三週沒動了」「你答應老闆下週一給更新，還剩兩天」「週五到了，要我幫你整理週報嗎？」

第三層的前提是第一層的 Living Work State——沒有持續追蹤的工作狀態，就沒有主動提醒的基礎。

### 核心元件設計

#### 1. 停滯偵測與主動提醒（Stagnation Detection & Nudges）

**設計概念：** Living Work State 中的每個任務都有「靜止天數」。當靜止天數超過閾值（預設 7 天，可由使用者調整），幕僚在下次對話中自然提及。

**提醒的語氣不是系統通知，是幕僚的觀察：**

| 停滯情境 | 幕僚的提醒方式 |
|:---|:---|
| 一般任務靜止 > 7 天 | 「順帶一提，API 標準化提案已經一週沒有更新了。是還在等架構委員會，還是有其他情況？」 |
| 有 deadline 的任務逼近 | 「你上週跟客戶承諾週三前給規格書，今天已經週一了。需要我幫你整理一下目前有的素材嗎？」 |
| 關鍵任務持續惡化 | 「build study 已經第三週了，每次提到它的語氣都比上一次更模糊。這可能是需要在週會上主動提出來的時機。」 |

#### 2. 週報與狀態報告自動生成（Automated Report Generation）

**設計概念：** 幕僚從一整週的日常對話中，自動組織出結構化的週報。使用者不再需要從零開始回憶「這週做了什麼」。

**週報的資料來源：**
- Living Work State 的變更歷史（新增 / 完成 / 狀態變更 / 阻塞）
- 日常對話中的關鍵事件記錄
- 待追蹤的承諾及其狀態
- 若有觸發第二層分析，包含分析摘要

**與 The King's Hand 的區別：** 週報是事實組織——「這週發生了什麼」。The King's Hand 分析是判斷——「報告裡隱藏了什麼」。兩者可以結合，但本質不同。

#### 3. 趨勢偵測與預警（Trend Detection & Early Warning）

**設計概念：** 當系統累積了同一專案的多次分析結果後，它能偵測趨勢——某個風險是在惡化還是在改善，某個補丁模式是在蔓延還是在收斂。

**實作方式：**

在 User Profile 或專案檔案中維護一個 **分析歷史摘要表**：

```markdown
## 專案分析歷史：Project Ares

| 週次 | 日期 | 狀態 | 關鍵發現摘要 | 風險趨勢 |
|:---|:---|:---|:---|:---|
| W10 | 2026-03-08 | 🟡 | DMA workaround 引入，PPA 影響未量化 | — |
| W11 | 2026-03-15 | 🟡 | DMA workaround 擴散到 3 個模組，功耗增加 15% | ↗ 惡化 |
| W12 | 2026-03-22 | 🔴 | 客戶 NordicTel 升級投訴，DMA 問題確認影響量產時程 | ↗↗ 急劇惡化 |
```

系統在每次新分析時，自動比對歷史摘要，在報告中增加「趨勢比較段」——明確指出哪些問題正在惡化、哪些已被解決、哪些是新出現的。

#### 3. 主動問題浮現（Proactive Issue Surfacing）

**設計概念：** 當系統在分析 A 專案時發現的模式，與 B 專案的歷史分析中出現過的危機前兆高度相似時，主動提醒使用者。

**範例場景：**
- 系統在分析 Project Mercury 時發現「韌體團隊開始繞過 code review 直接提交 hotfix」
- User Profile 中記錄了 Project Ares 在崩潰前 3 週也出現過完全相同的模式
- 系統在 Module 3 中主動加入：「此模式與 Ares 專案第 10 週的前兆信號高度一致。Ares 在此信號出現後 3 週進入紅燈。建議立即確認 Mercury 的 code review 流程是否仍在正常運作。」

### 第三層驗收標準

| 檢核項目 | 目標 | 驗證方式 |
|:---|:---|:---|
| 排程提醒準確性 | 在使用者開啟新對話時，正確提醒應執行的定期分析 | 模擬多週使用情境 |
| 趨勢偵測有效性 | 正確識別連續 3 次分析中的風險惡化趨勢 | 使用 King's Hand 現有測試案例模擬時序資料 |
| 跨專案模式匹配 | 當新專案出現已知危機前兆時，至少在 80% 的情況下觸發主動提醒 | 人工構造配對測試案例 |
| 無噪音原則 | 主動提醒的誤報率 < 20%（使用者認為無價值的提醒佔比） | 使用者回饋追蹤 |

## 開發藍圖：五階段漸進式演進

本計畫採用五個階段的漸進式開發路徑。與前一版六階段計畫的關鍵差異：Phase 0 和 Phase 1 合併重構，因為在新架構下，「分離核心與領域包」和「建立持續在場的夥伴基礎」是同一件事。

### 階段總覽

```
Phase 0 (基礎)  → 持續在場的夥伴核心：初識對話 + Living Work State + 日常更新吸收
Phase 1 (利刃)  → 分析兵器庫：敵對性分析作為技能 + 跨領域驗證
Phase 2 (進化)  → 持續進化：偏好學習 + 知識沉澱 + Domain Pack 自動生成
Phase 3 (主動)  → 主動式智慧：停滯偵測 + 期限預警 + 週報自動生成
Phase 4 (生態)  → 多使用者與社群：知識共享 + 多人協作
```

### Phase 0 — 持續在場的夥伴核心

**目標：** 建立系統的基礎身份——一位持續在場的工作夥伴。使用者能和幕僚進行初識對話、每天告訴他事情的進展、在需要時請他幫忙看一份文件。Living Work State 是核心交付物。

**交付物：**

| 項目 | 說明 |
|:---|:---|
| `SKILL.md` 通用化重構 | 從 The King's Hand 原始 SKILL.md 中抽離 IC 設計特定內容。定義四種運作模式（初識 / 日常 / 快速掃描 / 完整分析）的觸發條件與執行邏輯 |
| 初識對話引擎 | Step 0 雙模態設計的完整實作。包含問題生成（5 個方向）、回答→框架映射、Manager Profile 初始化 |
| Living Work State 機制 | 資料結構定義（任務表 + 事件日誌 + 承諾追蹤）。自然語言輸入 → 結構化狀態更新的解析邏輯 |
| 日常對話模式 | 使用者用自然語言告訴幕僚事情，系統在 `<thinking>` 中解析為狀態更新，維護 Living Work State |
| 快速掃描模式 | 使用者帶著具體問題提交片段文字，系統執行 Step 1 + Step 3 子集，以對話方式回覆 |
| 四層記憶架構 | Working Memory / Living Work State / User Profile / Domain Knowledge Base 的持久化機制 |
| IC 設計領域包分離 | 原有四份參考文件移入 `domain_packs/ic_design/` |
| IC 設計 16 案例回歸 | 透過「完整分析」模式觸發，維持 100% 通過率 |

**驗收標準：**

| 檢核 | 目標 |
|:---|:---|
| 初識對話 | 3-5 輪對話後，Manager Profile 準確反映使用者的工作脈絡 |
| 日常更新吸收 | 從 10 段自然對話中，正確擷取任務狀態更新 ≥ 90% |
| Living Work State | 經過 5 天模擬互動後，工作狀態表準確反映所有任務的最新狀態 |
| 快速掃描 | 使用者帶具體問題時，系統以對話方式提供針對性回覆 |
| **既有使用者零摩擦** | 擁有完整 Profile + Domain Pack 的使用者（如 Henry）丟文件進來時，系統不要求任何額外互動，不觸發初識對話，直接產出完整分析報告，體驗與原版 The King's Hand 零差異 |
| IC 設計回歸 | 16/16 測試案例通過 |

### Phase 1 — 分析兵器庫與跨領域驗證

**目標：** 在持續在場的夥伴基礎上，裝備 The King's Hand 的完整分析能力作為一項可按需觸發的技能。同時驗證系統在多個領域中的可用性。

**交付物：**

| 項目 | 說明 |
|:---|:---|
| 完整分析模式 | 五步推理鏈 + 五模組報告的通用化實作，由「完整分析」模式觸發 |
| Step 0.5 知識厚度自評 | L1/L2/L3 分級、Top-Down / Bottom-Up 動態比例 |

## 安全與治理原則

### 不可妥協的原則

1. **使用者資料主權：** User Profile、Living Work State 完全歸使用者所有。Living Work State 包含使用者的任務、承諾、工作進展等高度敏感資訊——系統永不在未經使用者同意的情況下共享這些內容。Domain Pack 的共享是 opt-in，且共享前會自動移除所有可識別個人或公司的資訊。

2. **分析誠實性：** 無論系統進化到何種程度，The King's Hand 的核心誓言不變——**「不會安慰你，不會替誰護短，不會在沒有證據時捏造紅燈。」** 偏好學習可以改變報告的風格與表達方式，但不可以改變分析結論的客觀性。

3. **偏好 ≠ 偏見：** 如果使用者的偏好實際上會導致盲點（例如偏好忽略某類風險信號），系統應在遵循偏好的同時，在報告中明確標注「此分析已根據您的偏好設定，省略了以下類型的信號，建議定期進行無偏好的完整分析。」

4. **動態擴充的安全邊界：** 當部署環境涉及程式碼執行時（如 Claude Code 環境中的自動化腳本），所有動態生成的腳本必須在沙盒環境中執行。本計畫認同「AI 助手進化開發計畫」中對 e2b MicroVM 沙盒的評估——當系統演進到具備自主程式碼生成能力時，硬體級隔離是唯一可接受的安全邊界。在此之前，文件系統層級的隔離（只讀掛載 + 輸出目錄白名單）已足夠。

5. **漸進信任模型：** 系統的自主權隨使用者信任的建立而逐步擴大。初始狀態下，所有知識庫變更都需要使用者確認。經過穩定的互動歷史後，低風險的變更（如新增術語翻譯）可自動執行，但高風險變更（如修改覆寫觸發規則）始終需要人工確認。

## 與「AI 助手進化開發計畫」的關係：務實的採納策略

「AI 助手進化開發計畫」是一份極具深度的技術研究報告，提出了分層圖譜記憶、MicroVM 沙盒、多代理拓撲、主動式心跳監控、CIPHER 隱性偏好學習等先進架構。本開發計畫對這些技術的採納態度如下：

| 技術提案 | 本計畫態度 | 理由 |
|:---|:---|:---|
| **分層圖譜記憶（HMLR / Event Progression Graph）** | Phase 2 概念採納，Phase 5+ 完整實作 | 概念上正確，但在 Claude Agent Skill 的文件系統環境中，結構化 Markdown 是務實的起步方案。遷移到圖資料庫的路徑已預留 |
| **e2b MicroVM 沙盒** | Phase 3+ 條件採納 | 當系統具備自主程式碼生成能力時導入。在此之前，文件系統隔離已足夠 |
| **RepoAudit 程式碼審計** | Phase 3+ 條件採納 | 當系統開始自動生成 Domain Pack 中的處理腳本時導入 |
| **多代理拓撲（Architect / Developer / Auditor）** | Phase 5+ 長期願景 | 當前的單一代理架構在分析任務中已經足夠強大。多代理的價值在於自主軟體開發——這是更遠的目標 |
| **CIPHER 隱性偏好學習** | Phase 2 核心採納 | 核心概念直接可用——比較原始輸出與使用者修改，推斷描述性偏好。不需要模型微調，完全在提示詞層面實作 |
| **心跳監控系統** | Phase 4 採納 | 在具備排程基礎設施的部署環境中實作。在 Claude.ai 對話環境中，以「開啟對話時的主動提醒」替代 |
| **Webhook 事件驅動** | Phase 4+ 條件採納 | 需要 MCP Server 或自託管服務作為基礎設施 |
| **Agent Audit 安全分析** | Phase 3+ 條件採納 | 與 e2b 沙盒同步導入 |

**核心原則：** 每一項技術的採納都必須經過「在當前環境中是否可實作」與「對使用者是否產出即時價值」的雙重檢驗。學術上的正確性不等於工程上的優先級。先讓第一層和第二層穩定運作、產出真實價值，再逐步疊加更先進的基礎設施。

## 命名哲學

**為什麼叫「Make Everybody King or Queen」？**

The King's Hand 的原始設計是為一位特定的「國王」——Henry——服務。他是 IC 設計公司的高階主管，他需要的是半導體領域的敵對性情報。

但「看穿迷霧、直面真相」的需求不是國王的特權。

每一個在資訊不對稱中做決策的人——無論是管理一個五人新創團隊的技術長，還是監督一個建案的工地主任，還是評估一個臨床試驗進度的醫療主管——都值得擁有一位不會對你說「一切都好」的幕僚。

而這位幕僚不是一個被丟進文件就吐出報告的機器。他是一個每天都在你身邊的工作夥伴——你跟他說今天發生了什麼，他記住；你問他一份報告有沒有問題，他看穿；週五到了你不想整理週報，他幫你寫好了。他會隨著和你共事的時間越長而越懂你，但他永遠不會因為懂你就開始替你粉飾太平。

這個計畫的目標，是讓每一個使用者都成為自己領域中的國王或女王——不是因為他們擁有權力，而是因為他們身邊有一位忠於真相的幕僚。

> *「國王之手不為國王服務。國王之手為真相服務。當每個人都能觸及真相時，每個人都是國王。」*
