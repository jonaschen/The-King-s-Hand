## 先讀這段 — 這份計畫為何存在

The King's Hand 已經證明了一件事：當 AI 停止當一個禮貌的摘要機器，轉而扮演一個冷酷的敵對性分析師時，它能夠從一堆「一切正常」的專案報告裡，刺穿西瓜效應，挖出被掩埋的真相。

但 Henry 不是唯一需要這種能力的人。

一位醫院營運長需要知道，科室主任口中的「人力調配已優化」是否意味著護病比已經超標三個月。一位新創公司 CTO 需要知道，工程週報裡的「技術評估進行中」是否意味著核心架構已經走進死胡同但沒人敢說。一位供應鏈副總需要知道，供應商回覆的「交期略有調整」是否意味著產線即將斷料。

**西瓜效應不是半導體業的專利，它是所有矩陣組織、所有層級彙報結構的宿命。**

The King's Hand 的核心價值不在於它懂半導體，而在於它的分析架構：從原始資料中偵測矛盾、量化隱性成本、翻譯術語煙幕、產出無法迴避的靈魂提問。這套架構是可以通用化的。

本計畫的目標，是讓這套架構學會自我適應——讀懂使用者的領域、記住使用者的偏好、隨著互動持續進化——最終讓每一個人都能擁有自己的「國王之手」。

### Step 0 已建立的六元素領域框架

ROADMAP v1.6 的 Self-Adjustment Addendum 中，Step 0 定義了以下六個領域無關的分析元素，這是通用化的基石：

1. **Hard Constraint（硬性約束）**：該領域中不可違反的絕對限制（IC 設計中是物理定律與製程限制；醫療中是法規與患者安全；金融中是合規要求）
2. **Patch Pattern（補丁模式）**：該領域中「短期修補代替結構性修復」的典型手法
3. **Irreversible Milestone（不可逆里程碑）**：一旦錯過就無法挽回的關鍵節點（tape-out、產品上市、法規申報截止）
4. **Trade-off Axes（取捨軸）**：該領域的核心資源取捨維度（PPA、成本-品質-速度、風險-報酬）
5. **Tier-1 Relationship（一級關係）**：最重要的外部利害關係人（客戶、投資人、監管機構）
6. **Blame Transfer Seam（責任轉嫁接縫）**：組織中最容易發生責任推諉的分工邊界

**這六個元素構成了通用領域框架的完整參數空間。任何領域的分析，都可以被映射到這六個維度上。**

## 系統演進願景：三層自適應架構

基於 The King's Hand 的已驗證資產，以及「AI 助手進化開發計畫」中提出的進階架構研究，本計畫定義三層演進架構。每一層獨立可用，但層層疊加後產生倍增效果。

```
┌─────────────────────────────────────────────────────────────┐
│  第三層：主動式智慧（Proactive Intelligence）                │
│  ─ 心跳監控、事件驅動、排程執行                              │
│  ─ 不等使用者開口，在問題浮現前主動介入                       │
├─────────────────────────────────────────────────────────────┤
│  第二層：持續進化引擎（Continuous Evolution Engine）          │
│  ─ 隱性偏好學習、技能自動擴充、跨會話記憶                    │
│  ─ 每次互動都讓系統變得更聰明                                │
├─────────────────────────────────────────────────────────────┤
│  第一層：通用敵對性分析核心（Universal Adversarial Core）     │
│  ─ 領域自動推斷、框架動態建構、零配置啟動                    │
│  ─ 任何人丟進任何文件，立刻獲得有價值的分析                   │
└─────────────────────────────────────────────────────────────┘
```

## 第一層：通用敵對性分析核心（Universal Adversarial Core）

### 設計目標

使用者不需要知道「國王之手」是什麼，不需要做任何設定，只要把自己的專案報告、會議記錄、進度追蹤文件丟進來，系統就能自動判斷領域、建構分析框架、產出有洞見的報告與提問。

**最低可行體驗：** 一個醫療器材專案經理把他的 FDA 510(k) 審查進度報告貼進來，系統在零配置下產出一份報告，正確指出被掩蓋的法規風險，並產出三個讓團隊無法閃躲的問題。

### 核心元件設計

#### 1. 領域推斷引擎（Domain Inference Engine）

擴展 Step 0B 的領域識別能力，建立多信號融合的推斷機制：

**推斷信號源（依優先順序）：**

| 信號 | 權重 | 範例 |
|:---|:---|:---|
| 使用者明確聲明 | 最高（直接覆寫） | 「我是醫療器材公司的品保主管」 |
| Manager Profile 中的領域標記 | 高 | 歷史校準已確認為金融業 |
| 文件中的專業術語密度 | 中高 | 出現 tape-out / DRC / LVS → 半導體；出現 sprint velocity / burndown → 軟體開發 |
| 職稱與組織結構關鍵詞 | 中 | 「護理長」→ 醫療；「基金經理」→ 金融 |
| 度量衡與 KPI 類型 | 中 | dB / TOPS / nm → 半導體；AUM / Sharpe ratio → 金融 |
| 文件結構與格式慣例 | 低 | Jira CSV 格式 → 軟體/硬體開發 |

**推斷結果分類：**

- **已知領域（Recognized Domain）**：系統內建參考知識庫的領域（初始版本：半導體 IC 設計）→ 載入完整的領域專用參考文件
- **可推斷領域（Inferable Domain）**：系統能從文件推斷出六元素框架的領域 → 動態建構框架，標注信心等級
- **未知領域（Unknown Domain）**：信號不足 → 以通用框架分析，主動詢問使用者領域背景

#### 2. 動態框架建構器（Dynamic Framework Constructor）

當領域被識別後，系統在 `<thinking>` 中即時建構六元素框架。以下展示框架如何映射到不同領域：

**範例：軟體新創公司**

| 框架元素 | 映射內容 |
|:---|:---|
| Hard Constraint | 現金跑道（Runway）、關鍵客戶合約 SLA |
| Patch Pattern | 「先上線再說」的技術債、硬編碼（Hardcoding）客製邏輯、跳過測試直接部署 |
| Irreversible Milestone | 產品發布日、融資 Demo Day、大客戶 POC 截止日 |
| Trade-off Axes | 速度 vs. 品質 vs. 技術債 |
| Tier-1 Relationship | 領投 VC、旗艦客戶、關鍵技術合作夥伴 |
| Blame Transfer Seam | 前端 vs. 後端、產品 vs. 工程、新功能 vs. 維運 |

**範例：醫療器材公司**

| 框架元素 | 映射內容 |
|:---|:---|
| Hard Constraint | FDA/TFDA 法規要求、ISO 13485、臨床試驗倫理規範 |
| Patch Pattern | 以 Software Change Notice 迴避完整 V&V 流程、修改 IFU 而非修復設計缺陷 |
| Irreversible Milestone | 510(k) 送件截止、臨床試驗起始、PMA 審查時程 |
| Trade-off Axes | 安全性 vs. 上市速度 vs. 開發成本 |
| Tier-1 Relationship | FDA 審查員、KOL 臨床醫師、通路經銷商 |
| Blame Transfer Seam | R&D vs. RA/QA、臨床 vs. 工程、設計管制 vs. 專案時程 |

**範例：營建工程**

| 框架元素 | 映射內容 |
|:---|:---|
| Hard Constraint | 建築法規、結構安全係數、環評要求 |
| Patch Pattern | 先施工再補申請、用變更設計掩蓋原始設計缺失、將工期延遲歸因天候 |
| Irreversible Milestone | 建照核發、結構體完成、使用執照申請 |
| Trade-off Axes | 成本 vs. 工期 vs. 品質 |
| Tier-1 Relationship | 業主、主管機關、融資銀行 |
| Blame Transfer Seam | 設計 vs. 施工、總包 vs. 分包、監造 vs. 承攬 |

#### 3. 通用化推理鏈（Generalized Chain of Thought）

##### 結構性設計原則：Top-Down / Bottom-Up 動態比例

The King's Hand 在 IC 設計中的五步推理鏈幾乎是純 Top-Down 的——系統帶著成熟的框架（workaround taxonomy、corporate code words、override triggers）主動狩獵已知的異常模式。這能成功，是因為四個 Phase 的校準為框架注入了深厚的領域知識。

但通用化之後，框架是動態建構的，知識厚度不再有保證。一個剛從文件推斷出來的六元素框架和一個經過 16 個測試案例打磨的框架，有本質的差異。**如果系統對這個差異視而不見，用同樣的信心做分析，它會在陌生領域裡犯兩種錯：看到框架告訴它去找的東西（即使那東西不存在），以及漏掉框架沒有定義的關鍵信號。**

因此，通用化推理鏈必須在 Step 0 之後、Step 1 之前，增加一個**知識厚度自評（Knowledge Depth Assessment）**步驟。這個自評的結果將決定後續所有步驟中 Top-Down（框架驅動的主動狩獵）與 Bottom-Up（資料驅動的模式浮現）的比例。

##### Step 0.5：知識厚度自評（Knowledge Depth Assessment）

在 Step 0 完成領域推斷與框架建構後，系統必須立即在 `<thinking>` 中執行以下自評：

**自評維度：**

| 維度 | 評估問題 | 厚度指標 |
|:---|:---|:---|
| **框架來源** | 六元素框架是從已驗證的 Domain Pack 載入的，還是從當前文件即時推斷的？ | Pack 載入 = 厚；即時推斷 = 薄 |
| **補丁模式庫完整度** | 系統對該領域的「補丁模式」有多少具體實例？ | ≥ 5 個已命名模式 = 厚；僅有抽象描述 = 薄 |
| **術語辭典覆蓋度** | 系統能識別該領域多少「企業話術」到「真實含義」的映射？ | ≥ 10 條映射 = 厚；< 3 條 = 薄 |
| **覆寫觸發規則明確度** | 紅綠燈覆寫條件是否有具體、可驗證的規則？ | 有量化閾值 = 厚；僅有定性描述 = 薄 |
| **歷史校準深度** | 該領域是否有經過使用者回饋校準的分析歷史？ | ≥ 3 次校準 = 厚；首次接觸 = 薄 |

**自評結果分級：**

| 知識厚度等級 | 條件 | Top-Down : Bottom-Up 比例 | 系統行為調整 |
|:---|:---|:---|:---|
| **L3 — 深厚（Deep）** | 具備完整 Domain Pack + ≥ 3 次校準歷史 | 70 : 30 | 全力框架驅動狩獵。信心等級高。報告語氣果斷。 |
| **L2 — 中等（Moderate）** | 具備部分 Domain Pack 或 ≥ 5 次同領域互動累積 | 50 : 50 | 框架狩獵與資料浮現並重。分析結論附帶信心標注。報告中明確區分「框架驅動的發現」與「資料浮現的發現」。 |
| **L1 — 淺薄（Thin）** | 僅有即時推斷的框架，無 Domain Pack，無歷史校準 | 20 : 80 | 以資料為主導。系統的首要任務是讓文件自己說話——標記所有異常模式、矛盾信號、語氣突變，即使系統不確定這些信號的含義。框架僅作為組織輸出的骨架，不作為狩獵的指引。報告語氣審慎，大量使用「證據顯示」而非「系統判定」。靈魂三問側重開放式探索而非精確打擊。 |

##### 各步驟在不同知識厚度下的行為差異

| 步驟 | L3 深厚（IC 設計當前狀態） | L1 淺薄（陌生領域首次接觸） |
|:---|:---|:---|
| **Step 1 解構與降噪** | 用已知的異常關鍵詞庫主動掃描 | **不預設該找什麼。** 改為全文異常偵測：標記所有語氣突變（從樂觀轉為模糊）、所有數字與定性描述的矛盾、所有被動語態掩蓋的主語（「已被處理」→ 被誰處理？）、所有時間軸上的跳躍或空白 |
| **Step 2 價值鏈定位** | 用已知的價值鏈地圖精確定位 | **從文件中歸納價值鏈。** 識別文件中提到的所有利害關係人、交付物、依賴關係，自行建構該專案的價值鏈草圖 |
| **Step 3 矛盾偵測** | 用六元素框架系統性搜索已知模式 | **以矛盾本身為目標，不預設矛盾的類型。** 逐段比對聲明與證據，標記所有不一致之處。額外執行「缺席信號分析」：文件中刻意沒有提到什麼？哪些利害關係人從未被提及？哪些時間段沒有任何更新？ |
| **Step 4 影響外推** | 精確翻譯為已知的商業後果 | **保守外推。** 僅基於文件中明確存在的利害關係人與時間壓力進行推斷，不引入文件外的假設 |
| **Step 5 靈魂三問** | 三問皆為精確打擊，包含具體數字與最壞情境 | **防禦/問責問題正常提出（因為它直接源於文件中的矛盾）。規模外推與生態護城河問題調整為探索性提問**——目的是幫助使用者自己釐清這些維度，而非直接指控 |

##### 完整推理鏈結構（通用版）

| 步驟 | 名稱 | 核心任務 |
|:---|:---|:---|
| **Step 0** | 領域推斷與管理者校準 | 識別領域、建構六元素框架、載入 User Profile |
| **Step 0.5** | 知識厚度自評 | 評估框架的知識厚度，決定 Top-Down / Bottom-Up 比例，記錄於 `<thinking>` |
| **Step 1** | 解構與降噪 | 依厚度等級選擇「關鍵詞狩獵」或「全文異常偵測」模式 |
| **Step 2** | 價值鏈定位 | 依厚度等級選擇「精確定位」或「歸納建構」模式 |
| **Step 3** | 矛盾偵測 | 依厚度等級選擇「框架驅動搜索」或「矛盾浮現」模式 |
| **Step 4** | 影響外推 | 依厚度等級選擇「精確翻譯」或「保守外推」模式 |
| **Step 5** | 靈魂三問 | 依厚度等級選擇「精確打擊」或「探索性提問」模式 |

**知識厚度不是靜態的。** 每一次互動（使用者的回饋、追問、修正）都在增加厚度。一個 L1 的領域在經過 5-10 次互動後可能升級為 L2，最終固化為 L3。這就是第二層「持續進化引擎」的核心任務——持續把 Bottom-Up 浮現的知識沉澱為 Top-Down 可用的框架。

#### 4. 通用化五模組報告（Generalized 5-Module Report）

| 模組 | King's Hand 原版標題 | 通用版標題 | 內容調整 |
|:---|:---|:---|:---|
| M1 | 專案光譜定位與賦能價值 | **專案定位與戰略價值** | 價值鏈定位表格，連結到該領域的商業或組織目標 |
| M2 | 專案穿透性摘要與紅綠燈 | **穿透性摘要與狀態判定** | 紅綠燈機制不變，覆寫觸發條件由領域框架定義 |
| M3 | 跨層級協作摩擦力與風險 | **跨職能摩擦與隱性風險** | 敘事體分析不變，摩擦識別基於六元素框架中的 Blame Transfer Seam |
| M4 | 客戶/市場連動效應 | **外部連動效應** | 衝擊對象改為 Tier-1 Relationship，時間軸對齊 Irreversible Milestone |
| M5 | 國王的靈魂三問 | **靈魂三問** | 三類型不變，問題內容由領域知識填充 |

**知識厚度對報告的影響：**

報告結構在所有厚度等級下保持一致（五模組不變），但表達的信心程度與措辭會隨厚度調整：

| 模組 | L3 深厚 | L1 淺薄 |
|:---|:---|:---|
| **M2 狀態判定** | 直接覆寫官方狀態，語氣果斷：「系統判定為🔴紅燈」 | 提供證據但保留判斷空間：「基於以下矛盾信號，官方狀態可能被高估。系統傾向🟡黃燈，但此判定基於有限的領域知識，建議以下列證據為基礎自行評估」 |
| **M3 摩擦分析** | 精確命名補丁模式與責任歸屬，量化隱性成本 | 列出所有偵測到的異常信號與矛盾，但標注為「資料浮現的模式」而非「已確認的風險」。不強行歸類到特定的補丁模式 |
| **M5 靈魂三問** | 三問皆為精確打擊，包含具體數字與最壞情境 | 第一問（防禦/問責）基於文件中最明確的矛盾正常提出；第二、三問調整為引導使用者自行探索的開放式提問 |
| **報告尾部** | 無額外說明 | 新增「**分析基礎聲明**」段落：明確告知使用者本次分析的知識厚度等級、框架來源、以及哪些結論是高信心的、哪些是探索性的。同時建議使用者提供哪些額外資訊可以提升下次分析的精準度 |

## 第二層：持續進化引擎（Continuous Evolution Engine）

### 設計目標

系統不是一個靜態的分析模板——它是一個活的幕僚，越用越懂你。每一次互動都是一次學習機會：你改了它的報告措辭，它記住你偏好的表達方式；你追問了某個方向，它記住你關心的重點；你分享了一份新類型的文件，它擴展了自己的分析能力。

### 核心元件設計

#### 1. 分層記憶架構（Hierarchical Memory Architecture）

參考「AI 助手進化開發計畫」中的 HMLR 架構與事件演進圖概念，但務實地設計為 Claude Agent Skill 生態系統中可實現的版本：

**三層記憶模型：**

| 記憶層 | 存儲內容 | 持久性 | 實作機制 |
|:---|:---|:---|:---|
| **工作記憶（Working Memory）** | 當前對話的上下文、正在分析的文件內容 | 會話內 | LLM 上下文視窗（已有） |
| **使用者輪廓（User Profile）** | 領域、職位、關注重點、溝通風格偏好、領域特定術語表 | 跨會話持久 | `manager_profiles/{user}.md`，由 CALIBRATE.md 維護 |
| **領域知識庫（Domain Knowledge Base）** | 領域框架、補丁模式庫、術語辭典、歷史分析模式 | 跨使用者持久 | `domain_packs/{domain}/` 目錄下的結構化文件 |

**關鍵設計決策：為什麼不直接上圖資料庫？**

「AI 助手進化開發計畫」提出了圖結構記憶與 HMLR 等先進架構。這些在技術上是正確的方向，但在當前階段存在實際約束：Claude Agent Skill 的執行環境以文件系統為主要持久化機制，不具備常駐的圖資料庫服務。因此，本計畫採取「結構化 Markdown 文件 + 語意索引」的務實路線，在概念上模擬分層圖譜的效果：

- **實體追蹤**：在 User Profile 中以結構化表格維護關鍵實體及其狀態變化歷史（模擬節點與邊）
- **時序記憶**：以帶時間戳的事件日誌追蹤分析歷史（模擬時序圖）
- **語意去重**：CALIBRATE.md 在更新 Profile 時，負責合併重複資訊、解決矛盾（模擬園丁機制）

**進階路線圖**：當系統遷移到具備持久化服務的環境（如 MCP Server 或自託管服務）時，可無縫升級為真正的圖結構記憶。結構化 Markdown 是圖資料庫的序列化形式，遷移成本極低。

#### 2. 隱性偏好學習系統（Implicit Preference Learning）

參考 CIPHER 演算法的核心概念，設計為可在 Skill 生態系統中落地的版本：

**偏好萃取觸發時機：**

| 觸發事件 | 萃取內容 | 儲存位置 |
|:---|:---|:---|
| 使用者修改了報告的措辭 | 原始輸出 vs. 修改後結果 → 推斷表達風格偏好 | User Profile → `style_preferences` 區塊 |
| 使用者追問了某個模組的某個方向 | 追問方向 → 推斷關注重點優先序 | User Profile → `priority_order` 區塊 |
| 使用者否定了某個分析結論 | 否定原因 → 推斷領域知識修正 | Domain Knowledge Pack → 對應文件 |
| 使用者提供了系統未見過的文件類型 | 文件結構 → 推斷新的輸入格式處理規則 | Domain Knowledge Pack → `input_formats` |
| 使用者對靈魂三問表示某一類型特別有用 | 問題類型偏好 → 調整三問的配比策略 | User Profile → `question_preferences` 區塊 |

**偏好的透明性原則（White-Box Preferences）：**

所有被推斷的偏好都以自然語言儲存在 User Profile 中，使用者可以隨時檢視、修改或刪除。系統永遠不會建立使用者無法理解或無法控制的隱性模型。這不僅是技術選擇，更是信任的基石。

**CALIBRATE.md 的擴展：**

現有的 CALIBRATE.md 僅從單次對話記錄中萃取 Manager Profile。擴展後的版本將支援：

- **增量更新模式（Incremental Update）**：不再每次重建 Profile，而是比對新舊版本，僅合併新資訊
- **衝突解決邏輯**：當新的互動證據與既有 Profile 矛盾時，依據「近期行為 > 早期行為」「明確聲明 > 推斷偏好」的優先規則解決
- **信心衰減機制**：超過 N 次互動未被再次驗證的偏好，其信心等級逐步降低
- **跨領域遷移**：當偵測到使用者切換了工作領域，將通用偏好（如溝通風格、報告詳細度）遷移到新領域的 Profile 中，但不遷移領域特定知識

#### 3. 技能動態擴充機制（Dynamic Skill Expansion）

系統應能根據互動中遇到的新需求，自動擴充自己的分析能力：

**擴充觸發場景：**

| 場景 | 擴充動作 |
|:---|:---|
| 使用者反覆提交某種系統不擅長處理的文件格式 | 建立新的輸入解析規則，加入 Domain Pack |
| 使用者指出了系統遺漏的某類異常信號 | 將新信號加入領域框架的異常關鍵詞庫 |
| 使用者提供了系統不認識的術語翻譯 | 更新 Euphemism Dictionary |
| 某個領域的分析被反覆使用且品質穩定 | 將動態建構的框架固化為正式的 Domain Knowledge Pack |

**品質閘門（Quality Gate）：**

動態擴充必須通過品質閘門才能被持久化：

1. **一致性檢查**：新知識不得與既有知識庫矛盾（除非使用者明確覆寫）
2. **最小使用次數**：某個推斷偏好或新知識點必須在 ≥ 3 次互動中被使用且未被否定，才會從「候選」升級為「確認」
3. **使用者確認**：重大的知識庫變更（如新增完整的 Domain Pack）會在下次互動時簡要通知使用者並徵求確認

#### 第一層與第二層的結構性關係：知識沉澱迴圈

第二層的所有機制——偏好學習、記憶維護、技能擴充——在概念上做的是同一件事：**把 Bottom-Up 浮現的知識，沉澱為 Top-Down 可用的框架，推動知識厚度從 L1 向 L3 爬升。**

```
互動 1-3（L1 淺薄）
  ├─ Bottom-Up 主導：資料浮現異常模式
  ├─ 系統產出探索性報告
  └─ CALIBRATE.md 從互動中萃取：領域術語、使用者關注點、異常信號

互動 4-8（L2 中等）
  ├─ 候選知識通過品質閘門，升級為確認知識
  ├─ 補丁模式庫開始有具體實例（而非僅抽象描述）
  ├─ 術語辭典覆蓋度上升
  └─ Top-Down 與 Bottom-Up 比例趨近 50:50

互動 9+（邁向 L3 深厚）
  ├─ Domain Pack 累積足夠的已驗證知識
  ├─ 覆寫觸發規則有了量化閾值
  ├─ Top-Down 比例上升，分析精度與效率同步提升
  └─ 系統行為逐漸趨近 The King's Hand 在 IC 設計中的表現
```

**這意味著：通用化的系統不是 The King's Hand 的弱化版——它是 The King's Hand 的幼年形態。** 給它足夠的互動與校準，它在任何領域都能長成和 IC 設計版本同樣鋒利的武器。知識厚度自評（Step 0.5）確保它在成長過程中始終誠實，不會在尚未成熟時假裝自己已經成熟。

### 第二層驗收標準

| 檢核項目 | 目標 | 驗證方式 |
|:---|:---|:---|
| 偏好學習有效性 | 經過 5 次互動後，報告風格與使用者偏好的匹配度顯著提升 | 比較第 1 次與第 5 次輸出的使用者滿意度 |
| **知識厚度爬升** | 一個 L1 領域在 ≥ 8 次互動後升級為 L2，Step 0.5 自評結果正確反映升級 | 模擬多輪互動的縱向測試 |
| 跨會話記憶一致性 | 第 N+1 次對話能正確引用第 N 次對話中建立的知識 | 跨會話回歸測試 |
| 偏好透明性 | 使用者能在 Profile 中找到所有被推斷的偏好，且描述可理解 | 人工審查 Profile 內容 |
| 知識擴充品質 | 動態擴充的知識點不引入錯誤或矛盾 | 領域專家審查新增的 Domain Pack 內容 |
| 無退化原則 | 偏好學習不導致原有分析品質下降 | IC 設計 16 案例回歸測試 |

## 第三層：主動式智慧（Proactive Intelligence）

### 設計目標

從「等待使用者丟文件進來」轉變為「在問題浮現前主動提醒使用者」。這是最高階的自適應能力，需要第一層和第二層作為基礎。

### 核心元件設計

#### 1. 排程分析觸發器（Scheduled Analysis Trigger）

**設計概念：** 使用者設定定期分析排程（如「每週三下午，幫我分析最新的 Jira 匯出」），系統在指定時間自動啟動分析。

**實作路徑（依部署環境分級）：**

| 部署環境 | 實作方式 | 限制 |
|:---|:---|:---|
| **Claude.ai 對話** | 使用者每週手動啟動，但系統在 Profile 中記錄「上次分析時間」與「建議下次分析時間」，在使用者開啟新對話時主動提醒 | 無法真正的背景執行 |
| **Claude Code + Cron** | 透過 `crontab` 排程執行 `analyze_pptx.sh --manager {user}`，輸出結果存入指定路徑 | 需要本機環境 |
| **MCP Server 整合** | 長駐服務接收 Webhook（如 Jira 事件、Git push），自動觸發分析並推播通知 | 需要 MCP Server 基礎設施 |

#### 2. 趨勢偵測與預警（Trend Detection & Early Warning）

**設計概念：** 當系統累積了同一專案的多次分析結果後，它能偵測趨勢——某個風險是在惡化還是在改善，某個補丁模式是在蔓延還是在收斂。

**實作方式：**

在 User Profile 或專案檔案中維護一個 **分析歷史摘要表**：

```markdown
## 專案分析歷史：Project Ares

| 週次 | 日期 | 狀態 | 關鍵發現摘要 | 風險趨勢 |
|:---|:---|:---|:---|:---|
| W10 | 2026-03-08 | 🟡 | DMA workaround 引入，PPA 影響未量化 | — |
| W11 | 2026-03-15 | 🟡 | DMA workaround 擴散到 3 個模組，功耗增加 15% | ↗ 惡化 |
| W12 | 2026-03-22 | 🔴 | 客戶 NordicTel 升級投訴，DMA 問題確認影響量產時程 | ↗↗ 急劇惡化 |
```

系統在每次新分析時，自動比對歷史摘要，在報告中增加「趨勢比較段」——明確指出哪些問題正在惡化、哪些已被解決、哪些是新出現的。

#### 3. 主動問題浮現（Proactive Issue Surfacing）

**設計概念：** 當系統在分析 A 專案時發現的模式，與 B 專案的歷史分析中出現過的危機前兆高度相似時，主動提醒使用者。

**範例場景：**
- 系統在分析 Project Mercury 時發現「韌體團隊開始繞過 code review 直接提交 hotfix」
- User Profile 中記錄了 Project Ares 在崩潰前 3 週也出現過完全相同的模式
- 系統在 Module 3 中主動加入：「此模式與 Ares 專案第 10 週的前兆信號高度一致。Ares 在此信號出現後 3 週進入紅燈。建議立即確認 Mercury 的 code review 流程是否仍在正常運作。」

### 第三層驗收標準

| 檢核項目 | 目標 | 驗證方式 |
|:---|:---|:---|
| 排程提醒準確性 | 在使用者開啟新對話時，正確提醒應執行的定期分析 | 模擬多週使用情境 |
| 趨勢偵測有效性 | 正確識別連續 3 次分析中的風險惡化趨勢 | 使用 King's Hand 現有測試案例模擬時序資料 |
| 跨專案模式匹配 | 當新專案出現已知危機前兆時，至少在 80% 的情況下觸發主動提醒 | 人工構造配對測試案例 |
| 無噪音原則 | 主動提醒的誤報率 < 20%（使用者認為無價值的提醒佔比） | 使用者回饋追蹤 |

## 安全與治理原則

### 不可妥協的原則

1. **使用者資料主權：** User Profile 完全歸使用者所有。系統永不在未經使用者同意的情況下共享 Profile 內容。Domain Pack 的共享是 opt-in，且共享前會自動移除所有可識別個人或公司的資訊。

2. **分析誠實性：** 無論系統進化到何種程度，The King's Hand 的核心誓言不變——**「不會安慰你，不會替誰護短，不會在沒有證據時捏造紅燈。」** 偏好學習可以改變報告的風格與表達方式，但不可以改變分析結論的客觀性。

3. **偏好 ≠ 偏見：** 如果使用者的偏好實際上會導致盲點（例如偏好忽略某類風險信號），系統應在遵循偏好的同時，在報告中明確標注「此分析已根據您的偏好設定，省略了以下類型的信號，建議定期進行無偏好的完整分析。」

4. **動態擴充的安全邊界：** 當部署環境涉及程式碼執行時（如 Claude Code 環境中的自動化腳本），所有動態生成的腳本必須在沙盒環境中執行。本計畫認同「AI 助手進化開發計畫」中對 e2b MicroVM 沙盒的評估——當系統演進到具備自主程式碼生成能力時，硬體級隔離是唯一可接受的安全邊界。在此之前，文件系統層級的隔離（只讀掛載 + 輸出目錄白名單）已足夠。

5. **漸進信任模型：** 系統的自主權隨使用者信任的建立而逐步擴大。初始狀態下，所有知識庫變更都需要使用者確認。經過穩定的互動歷史後，低風險的變更（如新增術語翻譯）可自動執行，但高風險變更（如修改覆寫觸發規則）始終需要人工確認。

## 與「AI 助手進化開發計畫」的關係：務實的採納策略

「AI 助手進化開發計畫」是一份極具深度的技術研究報告，提出了分層圖譜記憶、MicroVM 沙盒、多代理拓撲、主動式心跳監控、CIPHER 隱性偏好學習等先進架構。本開發計畫對這些技術的採納態度如下：

| 技術提案 | 本計畫態度 | 理由 |
|:---|:---|:---|
| **分層圖譜記憶（HMLR / Event Progression Graph）** | Phase 2 概念採納，Phase 5+ 完整實作 | 概念上正確，但在 Claude Agent Skill 的文件系統環境中，結構化 Markdown 是務實的起步方案。遷移到圖資料庫的路徑已預留 |
| **e2b MicroVM 沙盒** | Phase 3+ 條件採納 | 當系統具備自主程式碼生成能力時導入。在此之前，文件系統隔離已足夠 |
| **RepoAudit 程式碼審計** | Phase 3+ 條件採納 | 當系統開始自動生成 Domain Pack 中的處理腳本時導入 |
| **多代理拓撲（Architect / Developer / Auditor）** | Phase 5+ 長期願景 | 當前的單一代理架構在分析任務中已經足夠強大。多代理的價值在於自主軟體開發——這是更遠的目標 |
| **CIPHER 隱性偏好學習** | Phase 2 核心採納 | 核心概念直接可用——比較原始輸出與使用者修改，推斷描述性偏好。不需要模型微調，完全在提示詞層面實作 |
| **心跳監控系統** | Phase 4 採納 | 在具備排程基礎設施的部署環境中實作。在 Claude.ai 對話環境中，以「開啟對話時的主動提醒」替代 |
| **Webhook 事件驅動** | Phase 4+ 條件採納 | 需要 MCP Server 或自託管服務作為基礎設施 |
| **Agent Audit 安全分析** | Phase 3+ 條件採納 | 與 e2b 沙盒同步導入 |

**核心原則：** 每一項技術的採納都必須經過「在當前環境中是否可實作」與「對使用者是否產出即時價值」的雙重檢驗。學術上的正確性不等於工程上的優先級。先讓第一層和第二層穩定運作、產出真實價值，再逐步疊加更先進的基礎設施。

## 命名哲學

**為什麼叫「Make Everybody King or Queen」？**

The King's Hand 的原始設計是為一位特定的「國王」——Henry——服務。他是 IC 設計公司的高階主管，他需要的是半導體領域的敵對性情報。

但「看穿迷霧、直面真相」的需求不是國王的特權。

每一個在資訊不對稱中做決策的人——無論是管理一個五人新創團隊的技術長，還是監督一個建案的工地主任，還是評估一個臨床試驗進度的醫療主管——都值得擁有一位不會對你說「一切都好」的幕僚。

這個計畫的目標，是讓每一個使用者都成為自己領域中的國王或女王——不是因為他們擁有權力，而是因為他們擁有真相。

> *「國王之手不為國王服務。國王之手為真相服務。當每個人都能觸及真相時，每個人都是國王。」*
